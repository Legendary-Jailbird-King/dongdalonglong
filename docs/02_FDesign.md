# 1 前言

本文档为"全栈外卖管理系统（Takeaway Management System）"的需求与功能设计规格书，基于项目开发计划书和项目源代码实现进行功能模块分析与详细设计说明。

| **软删除** | 通过设置 `is_deleted=1` 标记逻辑删除，数据物理保留，不影响历史订单外键 |

| **bcrypt** | 密码单向哈希算法，将用户密码加密为不可逆的哈希值存储 |
| **CORS** | Cross-Origin Resource Sharing，跨域资源共享，允许前端跨端口访问后端 API |

# 2 开发概要

## 2.1 开发的目的和背景

**开发目的：**

本项目旨在构建一套面向餐饮外卖行业的全栈信息化管理平台，覆盖顾客下单、商家接单管理、骑手抢单配送的完整业务闭环，实现外卖业务流程的数字化、自动化和实时化。

**开发背景：**

随着移动互联网和电子商务的快速发展，外卖配送已成为居民日常消费的重要场景。据行业数据显示，中国外卖市场规模已突破万亿元，用户规模超过5亿。然而，大量中小型餐饮商家仍依赖人工电话接单、纸质记录、口头指派骑手等低效方式运营，存在以下痛点：

- 订单信息传递延迟，顾客等待时间不可预估
- 商家与骑手之间缺乏高效的协同机制
- 订单状态不透明，顾客无法实时了解配送进展
- 软删除与历史数据追溯困难

本系统从零开发一套完整的外卖管理系统，针对性地解决上述痛点，实现：

- 顾客在线浏览菜品并完成下单
- 商家实时接收订单提醒并处理订单
- 骑手主动抢单并完成配送确认
- 全流程订单状态实时同步与完整追溯

**必要性与可行性：**

- **经济可行性**：系统采用开源技术栈（Python/FastAPI + Vue 3），开发成本低，后期维护成本可控
- **技术可行性**：前后端分离架构成熟，WebSocket 实时通信技术已在工业界广泛验证
- **法规合规**：系统遵循《网络安全法》《个人信息保护法》相关要求，密码加密存储，JWT 无状态认证保障用户数据安全
- **道德准则**：公平对待顾客、商家、骑手三方利益，不设置算法歧视机制，骑手抢单机制公开透明

**预期目标市场：**

本系统定位于中小型餐饮商家及区域性外卖配送服务商，产品原型计划于 2026 年 6 月完成并进入内部测试阶段。

## 2.3 系统构成

本系统采用 B/S 架构（浏览器/服务器模式），基于前后端分离设计，划分为三层：

| 层级 | 技术栈 | 职责 |
| --- | --- | --- |
| **表现层** | Vue 3 + Element Plus + Pinia | 用户界面：登录/注册、顾客端（移动端）、商家端（PC） 、骑手端（移动端） |
| **服务层** | FastAPI + SQLAlchemy + WebSocket | 认证鉴权、业务逻辑（状态机流转）、事务管理、WebSocket 实时推送 |
| **数据层** | SQLite（开发）/ MySQL（生产） | 数据持久化存储，7 张核心业务表 |

**系统组件结构及关联关系：**

```
app/main.py ── 应用入口：注册路由、CORS、生命周期事件
    │
    ├── routers/auth.py      认证路由（注册、登录，2 个端点）
    ├── routers/customer.py  顾客路由（浏览、下单、地址、订单，8 个端点）
    ├── routers/merchant.py  商家路由（菜品、订单、WS 端点，10 个端点）
```

### WebSocket 连接鉴权：

- 连接端点：`ws://host:8001/api/merchant/ws/{role}/{user_id}?token={jwt_token}`

- 前端断线自动重连：5 秒后自动重新建立 WebSocket 连接

### 安全性：

- JWT Token 有效期 24 小时，过期需重新登录
- WebSocket 连接需附带有效 Token，中间人无法伪造
- 前端 401 拦截器在 Token 过期时自动清除登录状态并重定向

# 4 实现方式

## 4.1 多端统一认证功能

### 4.1.1 功能概要

本功能模块实现顾客、商家、骑手三种角色在同一登录页面的统一注册与登录。系统根据用户选择的角色类型，将账户信息写入对应的数据库表（customers / merchants / couriers），并签发包含用户身份信息的 JWT Token。后端通过角色依赖注入确保各端 API 只能被对应角色的用户访问。

**用例图：**

```
                              ┌──────────┐
                   ┌──────────┤  新用户   ├──────────┐
                   │          └──────────┘          │
                   │ 注册                            │ 注册
              ┌────▼────┐                      ┌────▼────┐
              │ 顾客注册  │                      │ 商家注册  │
              │(customer)│                      │(merchant)│
              └─────────┘                      └─────────┘
                   │                                │
                   │                          ┌────▼────┐
                   │                          │ 骑手注册  │
                   │                          │(courier) │
                   │                          └─────────┘
                   │
              ┌────▼──────────────────────────────▼────┐
              │            登录系统 (Login)               │
              │    选择角色 + 输入用户名 + 输入密码         │
              └────────────────┬───────────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │   签发 JWT Token     │
                    │ (user_id+role+name) │
                    └─────────────────────┘
```

### 4.1.2 功能详细

**注册流程（活动图）：**

```
开始
  │
  ▼
用户选择角色 (customer / merchant / courier)
  │
  ▼
输入用户名 + 密码
  │
  ▼
前端发送 POST /api/auth/register {role, name, password}
  │
  ▼
后端根据 role 选择对应 Model (Customer/Merchant/Courier)
  │
  ├── 检查用户名唯一性 → [已存在] → 返回 409 "用户名已存在"
  │
  └── [不存在] → bcrypt 哈希密码
        │
        ▼
      创建账户实例 → 写入对应表
        │
        ▼
      返回 {user_id, name, role} → 前端提示"注册成功，请登录"
```

**登录鉴权流程：**

```
前端发送 POST /api/auth/login {role, name, password}
  │
  ▼
后端根据 role 选择查询对应表
  │
  ├── [用户不存在] → 返回 400
  ├── [密码错误] → bcrypt.verify 失败 → 返回 400
  └── [验证通过] → create_access_token(user_id, role, name)
        │
        ▼
      返回 {access_token, token_type, role, user_id, user_name}
        │
        ▼
      前端 Pinia auth store 持久化到 localStorage
        │
        ▼
      前端路由跳转到对应角色工作台

后续 API 请求:
  HTTPBearer → decode JWT → 校验 role → 查询 DB → 注入当前用户实例
```

## 4.5 WebSocket实时推送功能

### 4.5.1 功能概要

WebSocket 模块为商家端和骑手端提供实时消息推送服务。当订单状态发生变化时（新订单、订单取消、骑手指派），服务端通过 WebSocket 长连接向目标用户实时推送 JSON 格式通知消息，用户无需手动刷新页面即可获得最新订单动态。

### 4.5.2 功能详细

**连接生命周期：**

```
客户端                                    服务端
   │                                         │
   │── ws://host/api/merchant/ws/            │
   │   {role}/{user_id}?token=xxx            │
   │──────────────────────────────────────►  │
   │                                         │── 解析 Token 校验身份
   │                                         │── 校验 role 匹配
   │                                         │── [失败] → close(code=4003)
   │                                         │── [通过] → accept()
   │◄──── WebSocket 连接建立 (101) ────────   │
   │                                         │── 注册到 ConnectionManager
   │                                         │
   │── 心跳 (send text) ──────────────────►  │ (保持连接，防止超时)
   │                                         │
   │                                         │── 业务触发 push ...
   │◄──── {"type":"new_order", ...} ────────  │
   │                                         │
   │── 浏览器关闭 / 网络断开 ──────────────►  │── disconnect() 清理连接
   │                                         │
   │── (5 秒定时器到期) ───────────────────►  │── 重新建立连接
```

**ConnectionManager 设计：**

```python
_connections = {
    "merchant": { merchant_id: [ws1, ws2, ...] },
    "courier":  { courier_id:  [ws1, ws2, ...] },
    "customer": {}
}
```

- 按角色 + 用户 ID 两级索引管理连接，支持同一用户多设备同时连接
- 推送失败时静默跳过（try/except pass），不影响其他连接
- 断开时从对应 bucket 中移除

## 5.3 安全性

### 5.3.1 标准的符合性

系统符合《网络安全法》《个人信息保护法》《数据安全法》及 OWASP Top 10（2021）相关要求。关键安全措施：bcrypt 密码哈希、JWT + RBAC 三端鉴权、数据归属校验、SQLAlchemy ORM 参数化查询防注入。

**安全基准达成情况：** 身份认证（未携带 Token 返回 401）、访问控制（角色依赖注入 + 数据归属双重校验）、密码安全（bcrypt 单向哈希，日志不记录密码）、会话安全（JWT 24 小时过期，前端 401 自动清除）、最小权限原则（各角色权限严格隔离）、安全配置外部化（`.env` 注入，不入库）。

### 5.3.2 信息安全防护

**防信息泄露：** bcrypt 哈希存储密码、JWT 密钥外部化、CORS 白名单机制（仅放行 `CORS_ORIGINS` 配置域名）、所有资源操作前进行归属校验（订单 customer_id / 菜品 merchant_id / 配送 courier_id 匹配）、HTTPException 受控错误返回、Token 不写入 URL。

**防信息丢失：** 软删除（`is_deleted=1`）保证数据物理保留、下单使用 `async with db.begin()` 显式事务（异常自动 ROLLBACK）、Token 过期后前端 401 拦截器自动引导重登录。

**外部不当操作检测：** 未授权调用 → 401/403；跨角色越权 → 403；水平越权 → 403 归属校验失败；非法状态变更 → 400 状态机前置条件拦截；重复抢单 → 400 业务校验 + 数据库约束；WebSocket 未授权 → 关闭连接 code=4003。

**系统资源保护：** MySQL 模式 `pool_size=20, max_overflow=10, pool_pre_ping=True`；WebSocket `ConnectionManager` 按角色+用户ID 两级索引，断开自动清理；FastAPI 单进程无状态设计。

### 5.3.3 安全设施需求

**主要危险与防护：**

| 危险 | 严重程度 | 防护措施 |
| --- | --- | --- |
| 密码明文泄露 | 🔴 严重 | bcrypt 单向哈希（SM-01） |
| JWT 密钥泄露 | 🔴 严重 | 环境变量注入，`.env` 入 `.gitignore`（SM-02） |
| 订单数据不一致 | 🔴 严重 | 原子事务 `async with db.begin()`（SM-03） |
| 越权操作 | 🟠 较严重 | 数据归属校验（SM-04） |
| 竞态重复派单 | 🟠 较严重 | 业务前置校验 + 数据库约束（SM-05） |
| 误删除 | 🟡 中等 | 关键操作 ElMessageBox 确认（SM-06） |
| WebSocket 中间人攻击 | 🟡 中等 | Token 鉴权（SM-07） |
| Token 长期有效被盗用 | 🟡 中等 | 24 小时过期（SM-08） |

**必须遵从的安全规则：** 禁止明文密码、禁止手写拼接 SQL（全项目 ORM）、敏感配置外部化、API 端点必须鉴权（除注册/登录外）、资源操作必须归属校验、WebSocket 必须鉴权、前端敏感路由必须守卫。

### 5.3.4 数据安全与隐私

**传输安全：** 当前开发环境使用 HTTP/ws://，生产环境建议升级 HTTPS/wss:// + MySQL TLS。JWT Token 通过 HTTP Header 传递（WebSocket 限于协议通过 Query String）。

**存储安全：** 密码 bcrypt 哈希不可逆；JWT_SECRET 环境变量注入；建议 SQLite 文件设 `chmod 600`；日志不记录密码和 Token。

**备份策略：** 生产环境建议 `mysqldump` 每日全量备份，保留 7 天日备份 + 4 周周备份，异地存储，每月验证备份可用性。

**隐私保护：** 最小数据采集（仅用户名、地址、电话）；地址信息仅对相关订单的商家和骑手可见；三张独立用户表物理隔离；当前版本不支持账户注销（后续版本建议增加）。

**完整性策略：** 订单状态机严格流转（终态不可变更）、数据归属不可变、订单总额由 ORM 事件自动计算（API 不可写）、WebSocket 消息服务端生成。

**代码追溯矩阵（关键项）：** SM-01 密码哈希 [utils.py#L19-L27](backend/app/utils.py#L19-L27)；SM-03 下单事务 [customer.py#L175-L230](backend/app/routers/customer.py#L175-L230)；SM-04 归属校验 [customer.py#L162](backend/app/routers/customer.py#L162)、[merchant.py#L115](backend/app/routers/merchant.py#L115)、[courier.py#L114](backend/app/routers/courier.py#L114)；SI-01 状态机 [courier.py#L114-L116](backend/app/routers/courier.py#L114-L116)。

# 7 注意限制事项

## 7.1 制约／限制事项

| 编号 | 限制事项 | 详细说明 |
| --- | --- | --- |
| LN-01 | 单商家单骑手模型 | 当前版本每个订单仅由一个商家处理、一个骑手配送，不支持多商家拼单或多骑手接力配送 |
| LN-02 | 无支付集成 | 当前版本不包含在线支付功能，订单金额仅为展示用途，不涉及真实资金流转 |
| LN-03 | 无图片上传 | 菜品不支持图片展示，仅显示名称和价格 |
| LN-04 | 无地理位置追踪 | 骑手不支持实时位置追踪和距离计算 |
| LN-05 | 骑手抢单无分布式锁 | 高并发场景下可能存在竞态条件（同一订单被多个骑手同时抢到），当前版本通过数据库约束作为最后防线 |
| LN-06 | 浏览器兼容性 | 建议使用 Chrome/Edge 最新版本，不支持 Internet Explorer |
| LN-07 | 无分页机制 | 订单列表和菜品列表未实现分页，数据量大时可能存在性能问题 |
| LN-08 | 单服务器部署 | 系统设计为单实例部署，不支持分布式集群（WebSocket 状态存储在进程内存中） |
| LN-09 | 数据库文件存储 | 开发环境 SQLite 数据库文件存储在项目目录下，需注意备份 |

| 编号 | 假定事项 | 详细说明 |

## 8.2 术语表

| 术语 | 英文 | 说明 |
| --- | --- | --- |
| **订单状态机** | Order State Machine | 订单状态有序流转规则：0待处理→1已接单→2派送中→3已完成；分支：0→4顾客已取消、0→5商家已拒单 |
| **软删除** | Soft Delete | 通过标记字段（is_deleted=1）而非物理删除方式移除数据，保留数据用于历史追溯 |
| **原子事务** | Atomic Transaction | 数据库事务保证多个操作要么全部成功提交（COMMIT），要么全部回滚（ROLLBACK） |
| **WebSocket 推送** | WebSocket Push | 服务端通过 TCP 长连接主动向指定客户端实时推送 JSON 消息 |
| **JWT Token** | JSON Web Token | 包含 user_id、role、name、过期时间的签名令牌，用于无状态客户端认证 |
| **RBAC** | Role-Based Access Control | 基于角色的访问控制：通过 payload 中的 role 字段决定用户可访问的 API 范围 |
| **ORM 事件** | ORM Event | SQLAlchemy 提供的对象关系映射级事件钩子，在 INSERT/UPDATE/DELETE 时触发自定义回调 |
| **依赖注入** | Dependency Injection | FastAPI 的 Depends 机制，用于自动注入数据库会话和认证用户实例 |
| **复合索引** | Composite Index | 多列组合的数据库索引，加速多条件查询 |
| **连接池** | Connection Pool | 预创建并维护一组数据库连接，复用连接减少开销 |

| ⑧ | 骑手端→数据库 | 主动抢单，原子更新 orders + couriers |
| ⑨ | 骑手端→数据库 | 确认送达，orders.status=3，couriers.status=0 |

**订单状态码含义：**

| 值 | 状态名 | 说明 | 下一有效状态 |

### order_dishes（订单-菜品关联表）

| 属性 | 类型 | 说明 | 取值范围 | 备注 |
| --- | --- | --- | --- | --- |
| id | INTEGER | 主键 | 自增正整数 | PRIMARY KEY, AUTOINCREMENT |
| order_id | INTEGER | 所属订单 ID | 有效的 orders.id | FOREIGN KEY, ON DELETE CASCADE |
| dish_id | INTEGER | 菜品 ID | 有效的 dishes.id | FOREIGN KEY |
| quantity | INTEGER | 购买数量 | > 0 | NOT NULL, CheckConstraint("quantity > 0") |

> **注**：order_dishes 表是 orders 与 dishes 之间的多对多中间表，一条订单可有多种菜品，每种菜品有一个数量。ORDERS 级联删除时（CASCADE），关联的 OrderDish 记录自动删除。OrderDish 的任何 INSERT/UPDATE/DELETE 操作都会触发 ORM 事件，自动重新计算所属订单的 total 字段。
