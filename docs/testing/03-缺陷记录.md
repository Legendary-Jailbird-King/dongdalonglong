# 缺陷记录

## DEF-001 并发抢单缺少原子条件更新

| 字段 | 内容 |
|---|---|
| 严重级别 | P1 / 高 |
| 状态 | 待修复 |
| 关联需求 | R-DATA-02、R-COU-01 |
| 位置 | `backend/app/routers/courier.py` 的 `pickup_order` |
| 检出方式 | 自动化并发测试 |

**问题描述：** 抢单流程先读取订单状态和骑手状态，再修改并提交；查询没有行锁，也没有使用 `UPDATE ... WHERE status=1 AND courier_id IS NULL` 的原子条件更新。两个并发事务可能都通过校验并返回成功，最终后提交者覆盖 `courier_id`，同时两个骑手都变为忙碌。

**复现：**

1. 创建一个状态为 1、无骑手的订单。
2. 两个空闲骑手并发调用 `PUT /api/courier/order/{id}/pickup`。
3. 观察两个响应和最终骑手状态。

**期望：** 仅一个请求成功，另一个返回 400/409；只有获胜骑手变为忙碌。

**建议：** 使用带状态条件的原子 `UPDATE` 并检查受影响行数；MySQL 环境可配合 `SELECT ... FOR UPDATE`。订单更新和骑手状态更新必须位于同一事务。

## DEF-002 bcrypt 依赖未锁定导致注册和登录不可用

| 字段 | 内容 |
|---|---|
| 严重级别 | P0 / 阻断 |
| 状态 | 待修复 |
| 关联需求 | R-AUTH-01 |
| 位置 | `backend/requirements.txt`、`backend/app/utils.py` |
| 检出方式 | 首轮自动化执行 |

**问题描述：** `passlib[bcrypt]==1.7.4` 会安装当前最新 `bcrypt`。在 `bcrypt 5.0.0` 环境中，Passlib 初始化后端时触发 `ValueError: password cannot be longer than 72 bytes`，导致所有注册请求异常退出，登录也不可用。

**期望：** 按项目依赖文件全新安装后，注册和登录可正常使用。

**建议：** 在生产依赖中明确固定兼容版本，例如 `bcrypt==4.0.1`；后续再评估迁移到维护活跃的密码哈希方案。测试依赖已固定该版本，以便继续执行其余业务测试。
