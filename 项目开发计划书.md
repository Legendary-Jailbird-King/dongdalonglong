## 1. 项目概要

### 1.1 项目名

全栈外卖管理系统（Takeaway Management System）

### 1.2 开发目的和背景

**开发目的：**

本项目旨在构建一套面向餐饮外卖行业的全栈信息化管理平台，覆盖顾客下单、商家接单管理、骑手抢单配送的完整业务闭环，实现外卖业务流程的数字化、自动化和实时化。

**开发背景：**

随着移动互联网和电子商务的快速发展，外卖配送已成为居民日常消费的重要场景。据行业数据显示，中国外卖市场规模已突破万亿元，用户规模超过5亿。然而，大量中小型餐饮商家仍依赖人工电话接单、纸质记录、口头指派骑手等低效方式运营，存在以下痛点：

- 订单信息传递延迟，顾客等待时间不可预估
- 商家与骑手之间缺乏高效的协同机制
- 订单状态不透明，顾客无法实时了解配送进展
- 软删除与历史数据追溯困难

**必要性与可行性：**

- **经济可行性**：系统采用开源技术栈（Python/FastAPI + Vue 3），开发成本低，后期维护成本可控
- **技术可行性**：前后端分离架构成熟，WebSocket 实时通信技术已在工业界广泛验证
- **法规合规**：系统遵循《网络安全法》《个人信息保护法》相关要求，密码加密存储，JWT 无状态认证保障用户数据安全
- **道德准则**：公平对待顾客、商家、骑手三方利益，不设置算法歧视机制，骑手抢单机制公开透明

**预期目标市场：**

本系统定位于中小型餐饮商家及区域性外卖配送服务商，产品原型计划于 2026 年 6 月完成并进入内部测试阶段。

### 1.3 功能概要

**体系结构：**

本系统采用 B/S 架构（浏览器/服务器模式），基于前后端分离设计，划分为三层：

```
┌─────────────────────────────────────────┐
│  表现层 (Vue 3 + Element Plus)          │
│  · 顾客端：菜品浏览、购物车、下单、订单  │
│  · 商家端：菜品管理、订单处理            │
│  · 骑手端：抢单、配送确认               │
├─────────────────────────────────────────┤
│  服务层 (FastAPI + SQLAlchemy)          │
│  · 认证鉴权 (OAuth2 + JWT + RBAC)       │
│  · 业务逻辑 (状态机、事务管理)           │
│  · WebSocket 实时推送                   │
├─────────────────────────────────────────┤
│  数据层 (SQLite/MySQL)                  │
│  · 7 张核心业务表                       │
│  · ORM 事件触发器（总额计算）            │
└─────────────────────────────────────────┘
```

**功能概要：**

| 角色 | 核心功能 |
|---|---|
| **顾客（Customer）** | 浏览菜品（过滤已下架/已删除）、管理收货地址（软删除）、购物车下单（原子事务）、历史订单查询、取消待处理订单 |
| **商家（Merchant）** | 菜品 CRUD（含软删除与上下架）、订单列表/筛选、接单/拒单操作、空闲骑手查询、实时新订单 WebSocket 提醒 |
| **骑手（Courier）** | 可抢订单列表（status=1 且无骑手）、主动抢单（1→2，骑手空闲→忙碌）、我的配送订单、确认送达（2→3，骑手忙碌→空闲）、实时任务推送 |

**订单状态机：**

```
0 待处理 → 1 已接单 (商家操作) 或 4 已取消 (顾客操作) 或 5 已拒单 (商家操作)
1 已接单 → 2 派送中 (骑手抢单)
2 派送中 → 3 已完成 (骑手确认送达)
```

### 1.4 作业范围

**1.4.1 负责部分在产品整体中的定位**

本项目为完整的外卖管理系统全栈开发，从零构建整个系统。作业范围覆盖：

- 后端 API 服务全量开发（FastAPI + SQLAlchemy ORM）

### 5.2 认证方案

- **协议**：OAuth2 Password Flow + JWT Bearer Token
- **Token 有效期**：24 小时（可配置）
- **密码存储**：bcrypt 单向哈希（passlib）
- **权限模型**：RBAC（Role-Based Access Control），3 角色（customer / merchant / courier）
- **Token 验证流程**：

```
请求 → HTTPBearer 提取 Token → decode JWT → 校验角色 → 返回用户实例
                                              │
                                         403 ←┘ 角色不匹配
```

### 5.3 实时通信方案

- **协议**：WebSocket（基于 websockets 标准）
- **连接端点**：`ws://host:8001/api/merchant/ws/{role}/{user_id}?token=xxx`
- **鉴权方式**：Query String 传递 JWT Token
- **自动重连**：前端断线 5 秒后自动重连
- **消息推送场景**：

| 消息类型 | 推送方 | 接收方 | 触发条件 |
|---|---|---|---|
| `new_order` | 后端 | 商家 | 顾客下单成功 |
| `order_cancelled` | 后端 | 商家 | 顾客取消订单 |
| `order_assigned` | 后端 | 骑手 | 商家指派骑手 |

### 5.4 数据库技术方案

- **ORM 框架**：SQLAlchemy 2.0.35（异步模式）
- **数据库**：SQLite（开发，零配置）+ MySQL 8.0（生产，高并发）
- **连接池**：AsyncSession + pool_size=20
- **事务管理**：`async with db.begin()` 显式事务
- **触发器**：SQLAlchemy ORM Event（`after_insert` / `after_update` / `after_delete`）实现总额自动计算
- **软删除**：UPDATE `is_deleted = 1`，历史订单外键不受影响

### 5.5 前端技术方案

- **响应式设计**：Element Plus Grid + CSS Flexbox
- **移动端适配**：顾客端/骑手端 max-width: 600px 居中布局
- **PC 端适配**：商家端 max-width: 1200px 宽表格布局
- **路由守卫**：`beforeEach` 校验 `meta.role` + Pinia auth store
- **状态管理**：Pinia（auth store + localStorage 持久化）
